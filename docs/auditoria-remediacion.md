## Parte A — Resumen ejecutivo

### La paradoja del gate: verde y ciego

El gate del proyecto está **completamente verde**: `npm run typecheck` pasa sin
errores y `npm test` pasa **277/277 tests**. Y sin embargo la auditoría confirmó
**127 defectos** (8 críticos, 28 altos, 44 medios, 47 bajos) más **10 bugs de
interacción cruzada**. La conclusión de mayor nivel no es ninguno de los 127
hallazgos individuales — es esta:

> **El gate de tests no ejercita ni uno solo de los modos de fallo que importan.**
> Las 277 pruebas validan la *orquestación con stubs* (el patrón de inyección de
> dependencias del proyecto), pero las *integraciones reales* —donde viven todos
> los defectos— son "fronteras deliberadamente sin cobertura". El sistema puede
> reportar verde habiendo validado nada, y sus propios tests no lo notan.

Esto convierte el `npm test` verde en una **señal de falsa confianza**: es
condición necesaria pero radicalmente insuficiente. Toda corrección de este plan
debe entregarse **con un test de regresión que falle antes y pase después**.

### El veredicto de fondo

El sistema implementa una arquitectura ambiciosa con bases que **no sostienen sus
propias promesas**. Hay un patrón único y repetido detrás de la mayoría de los
defectos críticos: **el sistema confía en lo que el LLM *dice* por encima de lo
que *hay en disco*, y acepta como "verde" ejecuciones que no probaron nada.** El
lazo de valor que debería romper el problema de Goodhart (generar → revisar →
ejecutar → medir cobertura → aprender) está, en código, **estructuralmente
incompleto o anulado** en cada uno de sus eslabones.

## Parte B — Veredicto de salud de las features CORE (las piezas magistrales)

El owner definió las features core como las que: **(1)** analizan bien el código
en su conjunto, **(2)** desarrollan tests de calidad que aporten a la lógica de
negocio de forma agnóstica, **(3)** ejecutan esos tests aprovechando el feedback,
y **(4)** aprenden de cada ejecución acumulando conocimiento agnóstico de forma
*segura y determinista*. Estas debían ser intachables. Este es su estado real:

| Feature core | Veredicto | Killer findings (evidencia) |
|---|---|---|
| **1. Análisis del código / blast radius** | 🟠 **Roto en los casos que importan** | Los **merge commits** —el evento de deploy canónico— producen diff vacío (`repo-mirror.ts:66` usa `git show` plano), así que clasificación *y* cobertura quedan ciegas. El detector de lógica usa un conteo **neto** que deja que una rama nueva se esconda tras una eliminación no relacionada (`commit-classify.ts:104-125`). Es **JS/TS-céntrico**: cambios reales en Python/Go no se detectan. `analysis.json` se instruye al LLM a escribirlo pero **nadie lo lee de vuelta**. |
| **2. Generación de tests de calidad** | 🟠 **Frágil — el gate de valor puede anularse solo** | La orquestación confía en el `specs[]` **auto-reportado por el LLM** sin reconciliar contra disco (`opencode-client.ts:247`→`pipeline.ts:315/265`): decide no-op, dispara review y arma la lista de archivos del reviewer con lo que el LLM *imprimió*, no lo que *escribió*. En fan-out el resultado **se auto-aprueba** por la mera existencia de un archivo. |
| **3. Ejecución con feedback** | 🔴 **Roto — acepta verde vacuo** | Un reporte con **0 tests ejecutados** → `pass` (`playwright-report.ts:89-99`). Tests `skipped`/`timedOut`/`interrupted` → `pass` (`:73-79`). En code-mode, "no se recolectaron tests" (exit 0) → `pass` (`code-runner.ts:199-211`). El gate de ejecución, que debía ser objetivo, **deja pasar suites que no corrieron nada**. |
| **4. Aprendizaje seguro y determinista** | 🔴 **Inexistente en código** (`works=no`) | Los campos medidos **nunca se escriben** (cero asignaciones en `src/`, confirmado a mano). Engram es **solo una instrucción de prompt** — ningún módulo determinista llama `mem_save`/`mem_search`. La cobertura se **computa y se descarta** (nunca llega ni al manifest ni al `RunRecord`). En el path por defecto (diff/webhook) el manifest lo **escribe el LLM a mano**. No hay determinismo posible cuando el único mecanismo es prosa en inglés que el LLM puede ignorar. |
| **Keystone: change-coverage** (lo que haría reales 3 y 4) | 🔴 **Roto como ground-truth** | No-op por diseño contra deploys bundleados (`change-coverage.ts:269` + su test `:134`). Se **desactiva** si el agente omite un import (`fixtures.ts`) y **ningún gate** lo exige. **Sobre-reporta**: una vez que el wrapper V8 del módulo tiene `count>0`, marca toda línea cambiada como cubierta aunque la rama no se ejecute (`:242-249`). Se saltea en merge commits. **Nunca se persiste.** |
| **Seguridad de la auto-evolución** | 🔴 **Insegura por defecto** | El "outer guard" colapsa a **auto-merge propio** cuando no hay branch protection — que el código **nunca verifica** (`index.ts:615-624`). `SELF_MAINTAINER_AUTOMERGE` viene **ON** por defecto. El self-test pre-deploy corre **código del agente con los secretos del orquestador** en el entorno (RCE-adjacent). El swap no es atómico: hay una ventana sin recuperación armada. |
| **Determinismo (prioridad #1 declarada)** | 🔴 **Roto** | El namespace es **solo-SHA** y los volcados de cobertura están gitignored, así que **sobreviven a `clean -fd`** (sin `-x`). Correr el **mismo SHA dos veces** lee volcados rancios de la corrida anterior y puede dar un **veredicto de cobertura distinto** — exactamente lo opuesto a "las mismas tareas consecutivamente sin perder calidad". |

**Conclusión sobre las core:** ninguna de las cuatro cumple hoy su contrato de
forma determinista. La #3 y la #4 (con su keystone) —el corazón de la propuesta
de valor— están rotas en el sentido más peligroso: **reportan verde habiendo
medido nada**. Estabilizar estas piezas es la prioridad absoluta; cualquier
feature nueva construida sobre esta base hereda el verde falso.

## Parte C — Temas raíz transversales (arreglar la causa, no 127 síntomas)

Los 127 defectos colapsan en **cinco causas raíz**. Atacarlas resuelve familias
enteras de hallazgos a la vez.

### T1 · La palabra del LLM por encima del disco
El orquestador trata la salida del LLM como autoridad: `specs[]` auto-reportado
decide no-op/review (`generation`), el manifest lo escribe el LLM en el path por
defecto (`learning-memory`), `analysis.json` y la memoria engram "persisten" solo
si el agente obedece prosa, y el rail anti-coerción de la continuación es **texto
de prompt** sin verificación (`interactive-api`).
**Fix raíz:** el orquestador debe **reconciliar contra disco/git** (qué specs
existen realmente, qué objetivos/changeRef tiene el manifest) y **poseer toda
escritura de estado persistente**. La salida del LLM es a lo sumo un sanity-check.

### T2 · Fuga de estado entre corridas (no-determinismo del sustrato)
Namespace **solo-SHA** (`test-data.ts:11`) + volcados de cobertura **gitignored
que sobreviven `clean -fd`** (`repo-mirror.ts:59`, sin `-x`) + **un mirror por
repo** compartido entre corridas + el dir de cobertura **nunca se limpia** entre
ejecuciones. Resultado: el mismo SHA dos veces → datos y veredictos distintos.
**Fix raíz:** namespace por **`runId`** (único por corrida); `rmSync` del dir de
cobertura antes de cada `execute()` que se vaya a medir; limpiar cobertura
gitignored en el checkout. *Un solo cambio cierra 3-4 críticos/altos a la vez.*

### T3 · Infra mal clasificada como veredicto de calidad (en ambas direcciones)
Un fallo de spawn (ENOENT/OOM) se reporta como `invalid` → Issue en el repo
vigilado (`validate.ts`). Un DEV caído en apps sin `versionUrl` se reporta como
`fail` → Issue (`deploy-gate`/`pipeline`). Un `DeployTimeoutError` se registra
como incidente `qa-generator` → **dispara al maintainer** (cross-flow). Y la
inversa: una corrida vacía → `pass`.
**Fix raíz:** un clasificador **infra-vs-calidad** explícito y centralizado,
threadeado en validate/execute/code-runner/deploy-gate, con **fail-closed cuando
"no corrió nada"**.

### T4 · "Best-effort / fail-open" sin observabilidad = degradación silenciosa permanente
Patrón repetido: el error del reviewer hace fail-open y publica **sin revisar,
para siempre, con solo un `console.log`** (`pipeline.ts:268-271`); el auto-merge
falla y el PR **queda sin mergear para siempre** (`publish.ts:70-75`);
`containsSecrets` **alterna true/false** en llamadas consecutivas por el
`lastIndex` de un regex `/g` compartido (`sanitizer.ts:108`).
**Fix raíz:** todo fail-open debe estar **acotado** (tras K fallos → fail-closed)
y **visible en el artefacto** (cuerpo del PR/Issue, `RunRecord`), nunca solo en
consola. "Best-effort" sin telemetría es un fallo invisible.

### T5 · Brechas en la frontera de seguridad (paths de ejecución no confiable)
Code-mode corre el suite del repo vigilado **con el entorno completo del
orquestador, incluido `GITHUB_TOKEN`** (`code-runner.ts:223`) — código no
confiable con la credencial de push. El self-test del maintainer corre **código
del agente con secretos** en scope. El outer-guard colapsa a self-merge sin
branch protection. La invariante "el agente es read-only sobre repos vigilados"
se viola en el path de ejecución, no en el del agente.
**Fix raíz:** **scrub del entorno** a una allowlist en todo spawn de código no
confiable; exigir branch protection antes de cualquier merge autónomo; aislar la
ejecución del self-test.

## Parte D — Secuencia de remediación priorizada (el plan de acción)

El orden está diseñado para **estabilizar las bases primero**: detener el verde
falso y restaurar el determinismo, luego hacer real el lazo de valor, luego
endurecer seguridad/infra, y por último robustez y verdad documental. **Regla
transversal:** cada ítem se entrega con un test de regresión que falla antes /
pasa después (ver la paradoja del gate, Parte A).

### Fase 0 — Detener el verde falso y el verde vacío + restaurar determinismo
*La base de confianza. Sin esto, todo lo demás se construye sobre arena.*

1. **Ejecución fail-closed cuando no corrió nada** — ampliar el tipo `stats`
   (`playwright-report.ts:42`) y exigir ≥1 test ejecutado para `pass`; tratar
   `skipped/timedOut/interrupted` como no-pass; guard en `pipeline.ts` que
   rechace publicar con `run.cases.length===0`. *(Críticos #1, alto execute-2)*
2. **Code-mode fail-closed** — exigir >0 tests ejecutados por ecosistema (parsear
   el conteo; `pytest exit 5`/`go "no test files"`/`node --test tests 0` → no
   pass). *(Crítico #3)*
3. **Determinismo del namespace** — derivar `ns` de `runId` (no solo SHA) y
   `rmSync` del dir de cobertura antes de cada `execute()` medido; `clean -fdx`
   (o rm explícito) de `e2e/.qa/coverage` en `ensureMirror`. *Cierra de una vez:
   coverage rancio cross-run, colisión de datos en DEV mismo-SHA, y el merge
   ronda-1/ronda-2 del enforce.* *(pipeline-orchestration alto + cross-flow F-1/F-8)*
4. **Reconciliar specs contra disco** — listar los `*.spec.ts` realmente
   añadidos/modificados en `e2e/` (vía `git status --porcelain`) y usar **eso**
   como conjunto autoritativo para la decisión no-op y la lista del reviewer.
   *(Alto generation-1; raíz T1)*
5. **Diff de merge commits** — detectar SHAs multi-parent en `repo-mirror.ts` y
   usar `git show --first-parent` / `git diff <sha>^1..<sha>`. *Desbloquea
   clasificación y cobertura para el evento de deploy más común.* *(Alto repo-mirror/change-coverage)*
6. **Normalizar SHA en el deploy gate** — comparar short-vs-full
   (`deploy-gate.ts:56`); sin esto, toda app con `/version` de 7 chars hace
   timeout en cada corrida. *(Alto deploy-gate)*
7. **Hacer observable el no-op del keystone** — cuando hay líneas cambiadas y el
   proveedor V8 produjo entradas pero matcheó **cero** scripts, emitir un `WARN`
   ruidoso en la corrida en vez del benigno "unknown — 0/N". *(Crítico #2, primer
   paso; la solución completa —source maps— es Fase 1)*

### Fase 1 — Hacer real el lazo de valor (las piezas magistrales)
8. **Persistir el aprendizaje** — un escritor *del sistema* que tras
   execute/coverage haga upsert por id en el manifest: `coverage.files` desde
   `computeChangeCoverage`, `stability.runs/flakyRuns` desde la clasificación
   pass/flaky, `ledger` cuando un test antes-verde se pone rojo. Persistir la
   cobertura también en el `RunRecord`. *(Crítico #6, altos learning-memory)*
9. **Manifest determinista en el path por defecto** — que `runOpencode` también
   haga upsert del manifest desde el spec-set reconciliado + intent, y dejar de
   instruir al LLM a editarlo. *(Alto learning-memory)*
10. **Reviewer fail-open acotado y observable** — tras K errores consecutivos →
    fail-closed (Issue de infra, sin auto-merge); anotar en el PR/Issue
    "publicado sin review independiente"; añadir `parsed` a `ReviewResult` para
    no quemar rondas contra feedback no accionable. *(Medios review-loop; raíz T4)*
11. **Escalar por lógica AÑADIDA, no neta** — en `commit-classify`, escalar
    cuando cualquier archivo con comportamiento añade lógica; detección
    consciente del lenguaje (o señal débil de "línea fuente añadida bajo tipo
    no-generate"). *(Altos commit-classify)*
12. **Cobertura real para el keystone** — resolver URLs bundleadas vía source
    maps; honrar el anidamiento de rangos V8 (restar sub-rangos `count==0`);
    eslint `no-restricted-imports` que fuerce el import de `./fixtures`; en
    code-mode habilitar instrumentación (c8/`--cov`/`-coverprofile`). *(Críticos
    #2/#4, altos change-coverage)*
13. **Cerrar la promesa "commitea de vuelta"** — fallback a
    `mergePullRequest` (o un reconciliador que mergee en un poll posterior)
    cuando `enableAutoMerge` falla en repos vigilados; lease explícito o rama
    única por corrida para no pisar commits. *(Críticos #7/#8)*

### Fase 2 — Endurecimiento de seguridad e infraestructura
14. **Scrub de entorno en spawns de código no confiable** — allowlist (PATH,
    HOME, vars del lenguaje); quitar `GITHUB_TOKEN`/`OPENCODE_API_KEY`/
    `WEBHOOK_SECRET`/`QA_API_TOKEN` en `code-runner` (instalar y testear). *(Alto code-runner security; raíz T5)*

### ⭐ Flujo: CORE: Code mode (verify-mode) auto-detect/run/classify
**Estado:** `partial` · 8 hallazgos (4 crítico/alto, 4 medio/bajo)

Code mode auto-detects an ecosystem from repo marker files, installs deps, runs the repo's own suite, and classifies strictly by exit code (no flaky, no static gate) — and the unit-level orchestration in code-runner.ts is clean and well-tested. But the flow does NOT work correctly as a trust-preserving QA gate: it runs the WHOLE pre-existing suite (never scoped to the agent's new tests), the change-coverage keystone is structurally dead for code mode (no instrumentation is ever enabled, so it is always "unknown" and never blocks), and several common "no tests collected" exit codes (node --test, go, jest --passWithNoTests) return 0 and are reported as a real pass. Worse, the only non-Node runtime images live in the wrong container, so every non-Node app yields infra-error on every run; and the untrusted repo test command is spawned with the orchestrator's full env (GITHUB_TOKEN), breaking the read-only security boundary.

#### E·code-runner-1 — 🔴 CRÍTICO · correctness · Exit-code 0 from 'no tests collected' is reported as a real pass (false green)  _(confirmed)_
- **Evidencia:** code-runner.ts:199-211 classifies purely on `const ok = out.exitCode === 0; ... verdict: ok ? "pass" : "fail"` with zero distinction for empty/uncollected suites. I reproduced both empty-suite-exit-0 cases on this machine: `node --test` (the last-resort command at code-runner.ts:125) in a dir with zero test files prints 'tests 0 / pass 0' and EXITS 0; `go test ./...` (line 89) with no _test.go files prints '? gt [no test files]' and EXITS 0. No count/collection check exists anywhere in code-runner.ts.
- **Por qué rompe:** If the agent writes nothing the runner picks up (or the repo has no collected tests), the suite exits 0 → verdict 'pass'. With needsReview off, or with the reviewer approving (it reviews specs, not execution counts), the green flows to publishCode → auto-merge PR. The system reports a confident green having validated nothing, deterministically, on every consecutive run — exactly the Goodhart failure CLAUDE.md's value/trust section warns about. Note finding #2 confirms change-coverage cannot catch this for code mode either, so there is no backstop.
- **Fix:** Require >0 tests EXECUTED before allowing verdict='pass'. Per ecosystem: parse node --test 'tests N' (N===0 → skipped, not pass); treat pytest exit 5 (no tests collected) as skipped not fail; grep go output for 'no test files'; for jest/vitest do NOT pass --passWithNoTests and treat 0 collected as not-a-pass. At minimum gate the 'pass' verdict on a positive executed-test count parsed from the runner output.

### ⭐ Flujo: CORE: Commit classification (blast-radius decision)
**Estado:** `partial` · 6 hallazgos (2 crítico/alto, 4 medio/bajo)

classifyCommit (src/qa/commit-classify.ts) parses a Conventional-Commits header, derives changed files from the diff, and computes a "net logic added" signal to cross-check the message against the diff and escalate refactor/style/perf/chore/revert to generate when the diff adds logic. It is deterministic for identical input (verified), and the happy-path tests pass. #### E·commit-classify-1 — 🟠 ALTO · correctness · Repo-wide NET logic count lets a real new branch hide behind an unrelated removal (refactor not escalated)  _(confirmed)_
- **Evidencia:** src/qa/commit-classify.ts:104-125 netLogicAdded() sums added/removed across the WHOLE diff and returns `added - removed` (line 124); line 50 `hasLogicChange = netLogicAdded(diff) > 0`. - **Por qué rompe:** The contradiction escalation at line 58 fires only on hasLogicChange (net>0). Any refactor/style/perf/chore/revert that adds a new code path while deleting a comparable count of logic lines elsewhere nets to <=0, so a behavior-changing commit is silently classified regression/skip and its new path is never tested — the exact 'silently spends zero effort on a change that needs tests' failure.
- **Fix:** Escalate on ADDED logic, not net: track per-file (or per-hunk) added-logic counts and escalate when any added-logic > 0 in a behavior-bearing file. Root cause traced: `/(?:password)\\s*[:=]/i` cannot match `password\":` because the `\"` between `password` and `:` is neither whitespace nor `[:=]`. This is a broad false-negative across the entire egress surface (execute.ts logs→Issue, opencode-client diff→model, api.ts/chat.ts logs→chat) and is more impactful than the IPv6, audit-count, or pattern-count findings. Fix: allow an optional quote/whitespace between keyword and separator, e.g. `(?:api[_-]?key|token|secret|password|passwd|pwd)[\"']?\\s*[:=]\\s*[\"']?\\S+` (and mirror in generic-credential), plus a JSON-form test. — Otherwise no additional defects: the three egress boundaries (diff→model opencode-client.ts:303/767, logs→Issue reporter.ts:11, logs→chat api.ts:277 egress + chat.ts:84/buildRunContext ingress) are all genuinely wired through sanitizeText as the auditor's `state` claims, and sanitizeText itself is deterministic (uses .match()/.replace(), not .test() on shared /g regexes).

#### F-1 — 🟠 ALTO · Same-repo runs share ONE mirror dir whose gitignored coverage dumps survive clean -fd — a prior run's V8 dumps poison the next run's change-coverage keystone

#### F-4 — 🟠 ALTO · createRecord uses a millisecond-resolution id (run-<sha7>-<timestamp36>); two triggers for the same sha in the same ms collide on the PRIMARY KEY and the throw escapes on an un-awaited path, hanging the HTTP request
- **Evidencia:** src/server/history.ts:150 `const id = run-${sha.slice(0,7)}-${Date.now().toString(36)}` (PRIMARY KEY) ; src/server/runner.ts:39 createRecord called synchronously inside enqueueTrackedRun ; src/index.ts:744-762 continueRun → enqueueTrackedRun with NO try/catch, invoked from api.ts handleContinue inside the async server callback (index.ts:766) which has no surrounding try/catch and no unhandledRejection handler
- **Por qué rompe:** Two same-sha triggers landing in the same millisecond (a webhook redelivery + the operator clicking continue; a double-submit) produce identical ids. better-sqlite3's INSERT throws a UNIQUE-constraint error synchronously inside createRecord (history.ts:150 + the insert). On the webhook path enqueueApiRun (index.ts:101) is called from req.on('end') (index.ts:828) with no try/catch → the throw becomes an uncaught exception in the request callback. On the API path, handleContinue → continueRun (index.ts:744) → enqueueTrackedRun → createRecord throws inside the async createServer callback (index.ts:766) with no try/catch and no process unhandledRejection handler, so the POST /api/runs/:id/continue (and /api/runs) client never receives its 202 — the request hangs and the trigger is silently lost. The collision is a single-record bug, but the HANG is a cross-boundary failure: history's id scheme + the runner's synchronous createRecord + the server's unguarded async callback together turn a duplicate into a dropped, unobservable trigger.
- **Fix:** Make the id collision-proof (append randomBytes(4).toString('hex') or a monotonic counter to history.ts:150) AND wrap the createRecord/enqueue call in enqueueTrackedRun in try/catch that returns a clean 409/500 to the caller; add a top-level process.on('unhandledRejection') that records an incident rather than dropping the request.

#### F-5 — 🟡 MEDIO · Hardcoded 'qa-bot' in enqueueApiRun diverges from the configured testDataPrefix — cross-run orphan cleanup targets a namespace that never existed

For any app whose testDataPrefix != 'qa-bot' (a required field with no default — the four shipped configs happen to all use 'qa-bot', masking it), the cleanup pass at pipeline.ts:245 runs against 'qa-bot-<sha>' which the prior run never wrote, so the REAL orphans (prefixed with the configured value) accumulate on DEV forever. #### F-6 — 🟡 MEDIO · A run's verdict is written by TWO uncoordinated writers — the queue's finally-finalizer and the operator-cancel handler — so a cancel during execute() that does not abort still overwrites the cancelled record with the run's real verdict (and can publish)
- **Evidencia:** src/index.ts:738-742 cancelRun: queue.cancel() aborts the signal then writes verdict 'infra-error'/'cancelled by operator' ; src/server/queue.ts:30-34 cancel() only abort()s the AbortController — execute()/publish are not signal-aware mid-call ; src/server/runner.ts:96-103 the job's success path unconditionally updateRecord(verdict=run.verdict,status=done) after runPipeline returns ; pipeline.ts:164-166 checkSignal only fires at explicit checkpoints, NOT inside deps.execute or deps.publish
- **Por qué rompe:** Cancel and the run completion are two writers of the same RunRecord verdict with no precedence. cancelRun (index.ts:738) aborts the signal and immediately writes 'cancelled by operator'. But checkSignal (pipeline.ts:165) is only polled at discrete points (pipeline.ts:201,302,323,347,442); the abort cannot interrupt an in-flight deps.execute (Playwright run) or the publish call (pipeline.ts:497). So the pipeline runs to completion, returns a real verdict, and runner.ts:96-103's success branch overwrites the 'cancelled' record with 'pass'/'fail' AND, if pass, has already called deps.publish (pipeline.ts:497) — opening a PR for a run the operator cancelled. The order is a race: whichever of (cancel write) and (completion write) lands last wins, and the completion path also has the publish side effect. The single-funnel auditor flagged 'cancel does not abort execute', but the two-writer overwrite + the publish-after-cancel is the cross-flow consequence between api.ts cancel, queue.ts abort semantics, runner.ts finalizer, and pipeline.ts publish.
- **Fix:** Thread the AbortSignal into deps.execute/deps.publish (or re-check signal.aborted immediately before publish at pipeline.ts:496 and before the success updateRecord at runner.ts:96) and have the runner detect a record already finalized as cancelled and NOT overwrite it / NOT publish. #### F-9 — 🟡 MEDIO · currentRun() returns the newest enqueued/running record by timestamp, so under any backlog the TUI/SSE map the live run to the WRONG record — activity, cancel, and continuation can target a queued-but-not-executing run
- **Evidencia:** src/server/history.ts:91 `currentRunStmt = ... WHERE status IN ('running','enqueued') ORDER BY at DESC, rowid DESC LIMIT 1` ; queue.ts:16-28 processes FIFO (oldest enqueued first) ; index.ts:723 currentRun exposed via apiDeps ; runner.ts:50 a job flips to 'running' only when it leaves the queue
- **Por qué rompe:** The queue executes FIFO — the OLDEST enqueued job runs first (queue.ts:18 chains on tail). Anything keying off currentRun (a TUI 'what's running now' panel, deriving the live state label interactive-layer.md §3.2 calls a 'derived read-only state label') points at a job that has not started, so live activity/logs shown are for the wrong run, and an operator 'cancel current' could mark an enqueued job while the real DEV-touching run keeps going. The bug is in history's query but only bites once the queue (a different module) holds more than one item — a cross-flow ordering hazard between queue FIFO semantics and history's DESC ordering.
- **Fix:** Make currentRun prefer status='running' strictly (a separate query: WHERE status='running' ORDER BY at ASC LIMIT 1, only falling back to enqueued if none running), matching the queue's FIFO execution order. Add a test with a running + two enqueued records asserting currentRun returns the running one.
